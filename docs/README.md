# CULO Documentation

This folder contains the governing documents for CULO. Every major decision — product, partnership, community, automation, or direction — should be traceable to something written here.

---

## Document Index

### [/constitution](./constitution/)
**What it is:** The founding philosophy of CULO.
**Use it when:** You need to understand *why* CULO exists and what it stands for. The Constitution is the highest-level document. If a product decision conflicts with the Constitution, the Constitution wins.

### [/product-blueprint](./product-blueprint/)
**What it is:** The technical and functional specification of the CULO platform.
**Use it when:** You are building, extending or reviewing features. The Blueprint defines what CULO does, how it works, and what the experience should feel like for Publishers.

### [/business-rules](./business-rules/)
**What it is:** Commercial rules, pricing logic, partner programs, marketplace guidelines, subscriptions and notifications.
**Use it when:** You are making decisions about money, partnerships, access tiers or commercial relationships. Business rules should always be consistent with the Constitution.

### [/roadmap](./roadmap/)
**What it is:** Development priorities — what is being built now, what is planned next, and what is further out.
**Use it when:** You need to understand the current state of development, prioritise a new feature, or place a request in context.

---

## How to Use These Documents

**Philosophy question** → start with `/constitution`
**Feature or UX question** → start with `/product-blueprint`
**Pricing or commercial question** → start with `/business-rules`
**Development priority question** → start with `/roadmap`

When documents appear to conflict, defer to the one higher in this list. The Constitution is the source of truth for everything CULO is.

---

## Document Conventions

- All documents are written in Markdown and version-controlled with the codebase.
- Documents should be updated when the decisions they record change — not left to go stale.
- Sections that have not yet been written are marked as `[Coming Soon]`.
- Every document should be readable by a new developer, a new team member, or Claude in a fresh session without prior context.
